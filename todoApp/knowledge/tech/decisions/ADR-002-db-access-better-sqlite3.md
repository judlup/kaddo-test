# ADR-002 — Use better-sqlite3 as the database access layer

## Status

Accepted

## Context

The project persists data in a local SQLite file. Two candidates were evaluated for the database access layer:

- **better-sqlite3** — synchronous, low-level SQLite bindings for Node.js. Direct SQL, no ORM abstraction.
- **Drizzle ORM** — TypeScript-first ORM with a type-safe query builder and schema definition layer.

The project is an example application built to validate Kaddo's KDD workflow. The goal is to keep the technical foundation simple, readable, and aligned with the domain model defined in knowledge files — not to showcase ORM patterns.

## Decision

Use **better-sqlite3**.

## Reasons

- Synchronous API is a natural fit for a CLI tool where async complexity adds no value.
- SQL written in migrations and repositories is directly readable without knowing an ORM's query DSL.
- Lower abstraction level makes it easier to trace data flow from CLI command → service → repository → SQL — which aligns with the traceability goal stated in `codebase.md`.
- No schema duplication: the domain model is defined once in migration files and referenced directly in repositories.
- Keeps the dependency surface small — important for an example project where every dependency is a potential source of confusion.

## Consequences

- Queries are written in raw SQL inside repository functions.
- Schema changes require new migration files (never editing existing ones).
- The migration runner is implemented manually (no ORM migration tooling).
- Type safety for query results is handled via TypeScript interfaces in the model files, not generated by an ORM.
- If the project were to grow to require complex relational queries or schema evolution tooling, Drizzle ORM could be introduced as a future enhancement — but that is explicitly out of scope for v1.
